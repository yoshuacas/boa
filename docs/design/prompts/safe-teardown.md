Make `boa teardown` both **complete** and **agent-safe**. Two real bugs observed during end-to-end validation on 2026-04-24, documented in `docs/internal/FEATURE-QUEUE.md` as the "teardown leaves the DSQL cluster behind" item and the "agent bypasses teardown confirmation" concern. Treat these as one product concern: teardown should finish the job it promises, and should not be something an autonomous agent can invoke without a human.

**Bug 1 — incomplete teardown.** `cli/commands/teardown.mjs` reports success ("Teardown complete. Stack '<name>' has been destroyed.") but leaves the DSQL cluster alive. Root cause: the SAM template at `cli/templates/backend.yaml` sets `DeletionPolicy: Retain` and `UpdateReplacePolicy: Retain` on `DsqlCluster` (and on `UserPool` — same issue, also Retain + `DeletionProtection: ACTIVE`). When CloudFormation deletes the stack these resources are disassociated, not destroyed. `teardown.mjs` disables DSQL deletion protection (step 7) and Cognito deletion protection (step 8) but never issues `aws dsql delete-cluster` or `aws cognito-idp delete-user-pool`. The same likely applies to the S3 bucket (check the template — if Retain, it also survives). Net effect: every `boa init` / `boa teardown` cycle accumulates orphan DSQL clusters until the account hits the 20-cluster quota and new deploys fail. This actually happened and blocked validation today. Do not change `DeletionPolicy: Retain` on the SAM template — that guard is there to protect against accidental stack deletion via other tools (the CloudFormation console, a misbehaving deploy), and it is the right default. The fix is inside `teardown.mjs`: after `sam.remove()`, explicitly delete each Retained resource whose ID is in `config.json`. Then verify — confirm the cluster status is `DELETING` or gone, the user pool is gone, the bucket is gone — and fail loudly if anything survived. Silent partial cleanup is worse than a loud error.

**Bug 2 — agent bypass.** The current confirmation prompt ("Type the stack name to confirm deletion") only guards against human fat-fingers. It reads `refresh_token` via stdin, so `echo "<stack>" | boa teardown` satisfies it, and so does any coding agent running the command through a Bash tool — the agent has the stack name sitting in `.boa/config.json`. The guard should block autonomous-agent invocations while staying ergonomic for a human at a terminal. Two layers, both small: (1) a TTY check — `teardown.mjs` refuses to run when `process.stdin.isTTY` is false, with an error message like "boa teardown must be run interactively from a terminal." This alone kills most agent tool-call paths and piped invocations. (2) Harness-level deny — `cli/commands/init.mjs` writes `.claude/settings.json` on init to auto-approve `boa deploy` / `boa migrate` / etc. Add an **explicit deny** rule for `boa teardown` in the same file so Claude Code has to ask the human before running it. Together these cover TTY-less invocations and agent-approved invocations. Do not add browser-based confirmation, out-of-band OAuth, or a `--prepare`/`--execute` split for MVP — too much complexity for launch.

**Scope of work.**
- `cli/commands/teardown.mjs`: add TTY check as the first action (before any output); after `sam.remove()`, explicitly delete DSQL cluster and Cognito user pool (and S3 bucket if Retain-policy — verify against `cli/templates/backend.yaml`); add a verification pass that confirms each Retained resource is actually gone or in a deleting state; print failures loudly.
- `cli/commands/init.mjs`: when writing `.claude/settings.json`, include a deny rule for `boa teardown` alongside the existing allow rules. Look at what the file currently writes and add the deny entry in the same style.
- Consider whether the TTY/deny pattern generalizes to a reusable `cli/lib/` helper (something like `requireHuman()` or `requireInteractive()`). If yes, extract it — at least two future destructive commands (`boa remove <extension>`, a hypothetical `boa migrate --revert`) will need the same guard. If a shared helper feels premature, leave it inline in `teardown.mjs` — don't design for hypothetical commands that don't exist yet.
- `cli/__tests__/` (if there are tests for teardown — check): add coverage for the TTY-refusal path and the post-teardown verification.

**Do not touch.**
- The SAM template's `DeletionPolicy: Retain` on DSQL / Cognito / S3. Those protect against the wrong-tool-deletes-my-prod-db scenario, which is a real concern separate from teardown UX.
- The confirmation prompt ("Type the stack name to confirm deletion") itself — keep it as the inner guard for humans, even after the TTY check. Both layers.
- `pgrest-lambda` — this is purely a BOA CLI change.
- `boa remove <extension>` and `boa migrate` — do not retrofit the new guard onto them in this change. If you extract a helper, leave those commands for a follow-up.

**Validation.** After implementation, verify: (a) `echo "<stack>" | boa teardown` fails with a TTY error and no destruction happens, (b) an interactive `boa teardown` in a fresh test project actually removes the DSQL cluster and Cognito pool from the account (manual AWS CLI check), (c) a fresh `boa init` writes a `.claude/settings.json` whose rules deny `boa teardown`, (d) post-teardown the CLI reports which resources were verified gone.
